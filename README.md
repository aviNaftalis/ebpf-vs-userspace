# ebpf-vs-userspace

[![ci](https://github.com/aviNaftalis/ebpf-vs-userspace/actions/workflows/ci.yml/badge.svg)](https://github.com/aviNaftalis/ebpf-vs-userspace/actions/workflows/ci.yml)

**The experiment:** a process prints 200,000 log lines (one `write()` each). We
count the `ERROR` lines **while it runs**, four different ways, and measure how
much each way *slows the process down* versus running it unwatched.

| way to watch | how it works |
|---|---|
| **baseline** | nobody watching — the speed ceiling (defined as **1×**) |
| **eBPF** | a kernel hook on `write()` checks each line's first bytes; process untouched |
| **pipe \| grep** | pipe the output to `grep ERROR` running on another core |
| **strace** | `ptrace` — stop the process and inspect every `write()` from userspace |

All four count the same 20,000 `ERROR` lines correctly. The only question is cost.

## Results

![how much each watcher slows the process down](docs/img/results.png)

- **eBPF beats `strace` by ~100×.** `strace` context-switches to the tracer on
  *every* syscall; eBPF runs inside the kernel with no context switch.
- **But eBPF isn't free, and a plain `pipe | grep` can beat it.** eBPF runs on the
  process's own hot path every syscall; `grep` works in parallel on another core.
- eBPF still wins the two cases that matter most in practice: a process you **can't
  modify** (you can't pipe a black box), and **big log lines** ↓.

## When eBPF wins: bigger log lines

![cost vs log-line size](docs/img/sizes.png)

eBPF's cost is **per line** (it peeks 5 bytes and ignores the rest); `pipe | grep`
must **copy and scan every byte**. So small lines favor `pipe | grep`, but past
~16–32 KB per line eBPF wins. (Slowdowns read higher here than above because this
sweep's baseline writes to `/dev/null` — nearly free — so any watching is a bigger
multiple; the *shape* is the point.)

## The catch

eBPF can't *parse* — the kernel verifier bans unbounded loops and regex, so the
program only checks a fixed 5-byte prefix. **eBPF is for cheap watching/filtering,
not general computation.**

## Run it

Linux with BTF (`/sys/kernel/btf/vmlinux`) + root:

```bash
make && sudo ./scripts/bench.sh    # prints the table; writes results.csv + sizes.csv
python3 scripts/plot.py            # renders the charts into docs/img/
```

CI does all of this on `ubuntu-latest` and commits the refreshed charts + numbers
back into this README on every push.

Key files: [`error_count.bpf.c`](src/error_count.bpf.c) (kernel hook) ·
[`ebpf_observer.cpp`](src/ebpf_observer.cpp) (loader) ·
[`logtarget.cpp`](src/logtarget.cpp) (workload) ·
[`bench.sh`](scripts/bench.sh) · [`plot.py`](scripts/plot.py).

*Numbers are one CI runner's; the ratios and shapes are the point. Linux-only.*
