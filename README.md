# ebpf-vs-userspace

[![ci](https://github.com/aviNaftalis/ebpf-vs-userspace/actions/workflows/ci.yml/badge.svg)](https://github.com/aviNaftalis/ebpf-vs-userspace/actions/workflows/ci.yml)

**What does eBPF actually do better, and than whom?** Not log *parsing* — a native
binary beats it there, and the eBPF verifier won't even let you write a general
parser. eBPF's real edge is **transparently observing an unmodified process** and
**aggregating in the kernel** so events never get copied to userspace. Its main
rival for that job is `strace`/ptrace — and that's where eBPF wins big (tens to
hundreds of × in the table below). What it is *not* is "free": a probe that fires
on every syscall still costs something.

This repo proves it with one workload observed four ways. A target process emits
log lines (one `write()` syscall each); the task is to **count the `ERROR` lines
while it runs**:

| method | what it is |
|---|---|
| **baseline** | the target alone, unobserved — the speed ceiling |
| **eBPF** | hook `write()` in-kernel, check the line prefix, count — *target unmodified* |
| **pipe \| grep** | the normal `app \| grep ERROR` pipeline (cooperative userspace) |
| **strace / ptrace** | trace every `write()` from userspace — the thing eBPF replaces |

**What the numbers actually show** (and they surprised me — which is the point of
measuring):

- **eBPF vs `strace` is the real win** — tens-to-hundreds of × (see the table;
  it's noisy on a shared runner but always huge). `strace` uses ptrace, which
  traps to the tracer twice *per syscall* — at one tiny `write()` per line that's
  catastrophic. eBPF observes the same writes transparently for only ~2–3×.
  *If you must transparently trace an unmodified process, eBPF is the only sane option.*
- **eBPF is not free** — here it lands in the **same ballpark as `pipe | grep`,
  often a touch slower**. The tracepoint fires on every `write()` and the program
  does a `bpf_probe_read_user` per event (sub-µs each, but it adds up at this rate).
  For a *cooperative* stream you control, an old-fashioned `app | grep` is
  competitive — it doesn't intercept every syscall, and `grep` runs on its own core.
- **All observers report the correct count (20000)** — eBPF's accuracy isn't in
  question, only its per-event cost.

So eBPF's advantage isn't "cheapest at everything." It's: (1) **transparency** —
watch *any* unmodified process or kernel event, not just one that pipes to you;
(2) **in-kernel aggregation** — here it shipped *zero* bytes to userspace, just two
counters, whereas `pipe | grep` copies every byte out; (3) **reach** — packets,
kernel functions, things with no userspace equivalent. The price is a fixed
per-event cost and verifier-limited logic.

## The catch (why eBPF isn't a parser)

Look at [`src/error_count.bpf.c`](src/error_count.bpf.c): it only checks a
**fixed 5-byte prefix** (`"ERROR"`). That's not a stylistic choice — the kernel
**verifier** forbids what real parsing needs: unbounded loops, a big stack, heap,
regex. eBPF trades generality for safety and locality. So:

- ✅ **eBPF wins** at *transparent* observation and *in-kernel* filtering/
  aggregation — watching an unmodified process or kernel event and shipping only a
  summary to userspace. Against ptrace (`strace`) it's orders of magnitude faster,
  and it can reach events (packets, kernel functions) with no userspace equivalent.
- ❌ **eBPF is not automatically cheap.** A probe that fires on every syscall has a
  real per-event cost — for a *cooperative* high-rate stream, a plain `app | grep`
  is competitive or better. And it can't do *general computation* (rich
  parsing, transforms, anything unbounded); that belongs in a native binary
  (Docker/VM barely change that — they're isolation layers, not a different
  execution model).

The honest one-liner: **eBPF is the cheapest way to *transparently watch* (vs
ptrace) and to *aggregate in-kernel* — not the fastest way to *compute*, and not
free per event.**

## How it works

- [`src/logtarget.cpp`](src/logtarget.cpp) — the observed workload (N lines, one `write()` each).
- [`src/error_count.bpf.c`](src/error_count.bpf.c) — the in-kernel program on the `sys_enter_write` tracepoint; counts in a shared global, no per-event userspace copy.
- [`src/ebpf_observer.cpp`](src/ebpf_observer.cpp) — libbpf loader: load → attach → run the target → read the counters.
- [`scripts/bench.sh`](scripts/bench.sh) — runs all four methods, emits the results table + `results.csv`.
- [`scripts/plot.py`](scripts/plot.py) — renders `results.csv` into the graph (`docs/img/results.png`).
- CI (`.github/workflows/ci.yml`) builds it, runs the benchmark **as root on `ubuntu-latest`**, and commits the refreshed **graph + table** back into this README.

## Run it yourself

Needs Linux with **BTF** (`/sys/kernel/btf/vmlinux`) and **root**:

```bash
sudo apt-get install -y clang llvm libbpf-dev libelf-dev bpftool strace make
make
sudo ./scripts/bench.sh        # prints the Markdown results table
```

## Requirements & caveats

- **Linux only**, recent kernel **with BTF** (CO-RE); needs **root** (`CAP_BPF`/`CAP_SYS_ADMIN`).
- On **WSL2**: works if `/sys/kernel/btf/vmlinux` exists (recent 6.x kernels have it); otherwise use an Ubuntu VM/cloud box — which is what CI does anyway.
- Numbers are one runner's; the *ratios* (eBPF vs strace) are the point, not absolute values.
- A teaching benchmark, not a production tracer — the eBPF program favors clarity.
